# 開發歷程

這份紀錄整理「你猜猜看」從初版想法到目前 MVP 的產品演進。專案由使用者擔任發想者、產品審核者與方向決策者，Codex 作為協作開發者，依照回饋逐步調整實作。

## 1. 初始構想：默契問答網頁遊戲

最初的方向是一個單頁網頁遊戲，技術上要求使用純 `HTML + CSS + JavaScript`，不依賴後端，所有資料保存在瀏覽器本機。

初始需求包含：

- 內建 36 題
- 題目分類
- 新增自訂題目
- 隨機抽題
- 玩家 A 輸入真實答案
- 玩家 B 輸入猜測答案
- 揭曉答案
- 討論備註
- 猜中率、最常猜錯分類、收藏題目
- 匯出 / 匯入 JSON
- 深色模式
- HOSTER 模式

這個階段的產品想像比較接近完整網頁遊戲。

## 2. 模式設計：依熟悉度分層

使用者補充了四種模式，並指出模式的重點不是單純分類，而是遊玩對象彼此的熟悉度。

當時定義的方向：

- 朋友模式：輕鬆有趣，低侵入性
- 熟人模式：驗證彼此認知，中度探索
- 情侶模式：偏價值觀，深度對話
- 自訂題庫：保留給未來擴充

這一步讓專案從一般問答工具，轉向「依關係深度調整題目」的桌遊產品。

## 3. 題庫策略：先保留，未來再篩選

使用者提供了個人版 36 題，分為：

- Lv1：生活與習慣
- Lv2：個性與價值觀
- Lv3：人生故事
- Lv4：進階版

同時也明確指出，不需要直接棄用未被選進題庫的題目，候選題可以先保留。題庫最終決定權應保留在產品方，而不是開放玩家任意新增。

因此專案改成：

- 題庫獨立資料化
- 保留正式題包與候選題包
- 玩家端不提供新增題目入口

## 4. 重要轉向：網頁只是桌遊道具

開發過程中，使用者指出初版設計仍像是共用終端或表單系統，和真實遊玩情境不符。

使用者重新定義情境：

- 兩個人面對面玩
- 各自拿自己的手機
- 答案用口頭講
- 不需要輸入真實答案
- 不需要同步
- 不需要隨機抽題
- 不需要玩家新增題目
- 網頁只是幫助這個桌上遊戲的道具

這是產品方向最大的轉折。此後核心功能被收斂為：

- 列出題目
- 切換題號
- 留下對方猜測的評分與評語

## 5. 介面簡化：移除多餘功能

使用者持續審核畫面，指出多個區塊會讓產品變得像分析系統或後台工具。

因此移除或下架：

- 玩家 A / 玩家 B 答案輸入區
- 揭曉答案流程
- 隨機抽題按鈕
- 題庫管理區
- 題庫策略說明區
- HOSTER 遊戲中切換
- 猜中 / 未猜中二元評分
- 猜中率
- 最常猜錯分類
- 收藏題目
- 額外的同步玩法說明
- Mode A / B / C 這類突兀標籤

保留下來的是更像桌遊輔助卡的流程。

## 6. 評分方式：從真假改成 0 到 100

使用者指出評分不應該只是 `true / false` 或「猜中 / 未猜中」。猜測可能有程度差異。

因此評分改成 0 到 100 分的準確度滑桿，讓玩家可以記錄：

- 對方完全猜錯
- 猜到一部分
- 猜對方向但原因錯
- 幾乎完全理解

這也更符合「彼此認知」這個產品核心。

## 7. 單一對象紀錄

使用者指出，一次遊戲通常只會和一位對象玩，不需要在同一場遊戲內管理多個對象。

因此紀錄模型改成：

- 只保留一組「這次和誰玩」
- 同一題再次儲存時更新原紀錄
- 結束後可匯出當次遊戲結果

這讓資料模型更貼近真實使用情境。

## 8. 規則頁

使用者希望加入規則介紹頁，並在審核後補充第 5 點：

> 遊戲結束時，如果彼此都願意，可以交換手機看看彼此的筆記。

目前規則頁聚焦在：

- 選模式
- 對題號
- 口頭猜
- 留評語
- 交換筆記

## 9. 視覺方向：溫馨、可愛、淡色系

使用者不滿意早期 CSS，要求整體風格更溫馨可愛，且偏淡色系。

過程中曾嘗試自製插圖，但使用者指出品質不佳，因此改用授權更清楚、風格一致的線上 SVG 插圖。

後續又補充：不要只做深色模式，而是可以做「風格切換」。因此目前加入四種視覺樣板：

- 溫馨桌遊
- 朋友派對
- 情侶約會
- 復古題卡

這些樣板保持同一套版型，只變化配色與插圖。

## 10. 深色模式下架

曾經保留深色模式，但使用者指出深色效果不好，與產品氣質不合。

後來決定：

- 深色模式先關掉
- 不再提供黑白切換
- 改以風格樣板作為未來視覺擴充方向

## 11. RWD 調整

使用者使用裝置模擬檢查畫面，陸續指定：

- 手機以 375px 寬度規劃
- 平板另外切 iPad 尺寸斷點
- 桌機尺寸也要重新調整

目前已有三種主要版型：

- 手機：以 375px 為基準
- 平板：721px 到 1180px
- 桌機：寬版容器與多欄卡片

## 12. Git 與部署

專案建立 Git 後，使用者要求：

- Push 由使用者決定
- Codex 不准自行 push
- feature branch 完成後再 merge 到 main

目前 `feature/two-device-mode` 已經合併回 `main`，但是否 push 到 GitHub 仍由使用者決定。

GitHub Pages 部署方向：

- 使用 `main`
- 使用根目錄 `/`
- 由 `index.html` 作為入口

## 13. 題庫候選來源庫

使用者提出要開始優化題庫，但不要直接改程式碼或正式題庫。於是新增 `SOURCE/` 資料夾，作為產品方篩選用的候選題來源庫。

這一階段的分工是：

- 使用者：題庫方向發想、審核與最終入庫決策
- Codex：整理公開題型研究、生成候選題、建立穩定題號與檔案結構

目前 `SOURCE/` 依七個類型各整理 100 題：

- 生活習慣
- 興趣嗜好
- 工作與學習
- 消費觀
- 人際關係
- 愛情觀
- 人生觀

候選題不會被遊戲程式直接讀取。未來使用者可以依題號指定調整、改寫或入庫。

後續使用者希望再擴充幾類，但不要動正在審核中的既有七類，因此新增第二批候選分類：

- 自我認識
- 家庭與成長
- 情緒壓力
- 界線與衝突

這批題目同樣只作為 SOURCE，不進入正式遊戲題庫。

## 14. 介面改為 L1 / L2 題庫瀏覽

使用者決定新開分支，將既有 SOURCE 候選題整合進介面，讓玩家不只使用少量核心題，而能自行選擇想聊的題目類型。

新的介面邏輯改為兩層：

- L1：親密關係定義，包含朋友模式、熟人模式、情侶模式
- L2：題目類型，來自 SOURCE 的分類檔案

這個調整保留「讓玩家更熟悉彼此」的產品目的，同時讓題庫探索更自由。SOURCE markdown 仍作為審核原始資料，另轉出 `source-questions-data.js` 供前端讀取。

## 15. 建立匿名設計原型

使用者希望開始額外輸出原型，用來和外部設計協作者討論 UI 方向，但不暴露正式網頁標題與題目內容。

因此新增 `prototypes/` 作為獨立匿名原型區，只保留操作結構、視覺節奏與可替換保留位，不載入正式題庫或正式遊戲程式。原型可以直接用 HTML/CSS 編輯，也可以從瀏覽器輸出 PDF 或 PNG，交給一般美術人員標註配色、插圖、字體、間距與情緒方向。

這一階段先提供三個可編輯方向：

- 溫暖桌卡：偏桌遊與朋友聚會
- 手帳貼紙：偏可愛、親密與留下回憶
- 清爽活動卡：偏活動包、主持與未來商業擴充

使用者仍是審核與發想角色，Codex 依照「可交接給設計協作者修改」的目標整理原型檔案。

後續使用者提出希望把現有風格版型額外輸出成 `indexX.html`，方便各別輸出 PDF / PNG。於是新增：

- `prototypes/index1.html`：溫暖桌卡單頁
- `prototypes/index2.html`：手帳貼紙單頁
- `prototypes/index3.html`：清爽活動卡單頁

使用者接著釐清，真正需要的是把先前已設計過的產品風格做成匿名 sample，甚至可以給 Sales 展示，而不是抽象設計稿。  
因此另外新增 `samples/`，以目前產品風格為基礎輸出四個匿名展示頁：

- 溫馨桌遊
- 朋友派對
- 情侶約會
- 復古題卡

這些 sample 隱藏正式標題、真實題目與題庫資訊，只保留商品氣氛、卡片結構、互動元件與銷售展示情境。

使用者同意將 Codex 設計的插圖放入 sample，因此四個匿名 sample 進一步加入專案內 inline SVG 插圖，讓頁面更接近可展示稿，同時不依賴外部圖片來源。

後續使用者指出 sample / prototype 和實際產品 UI 層級仍有落差。Codex 判斷確實應拆成兩種 sample：

- `style-samples/`：原本的風格展示稿，展示商品氣氛與美術方向
- `product-samples/`：新增的產品 UI 展示稿，貼近實際 `index.html` 的操作結構與畫面層級

因此將原 `samples/` 重新命名為 `style-samples/`，並新增 `product-samples/` 四個匿名產品畫面，分別對應目前已有的四種視覺風格。

使用者詢問 `assets/` 是否自始至終為空。檢查 Git 歷史後確認，早期曾有 `assets/table-cards.svg`，後來因流程簡化被移除。依照使用者新的整理規則，未使用 SVG 應封存在 `assets/deprecated/`，因此將舊的桌卡 SVG 從歷史中取回並保存為 `assets/deprecated/table-cards.svg`。

## 目前產品定位

目前版本不是完整商業化產品，而是一個可部署、可驗收的 MVP：

- 輔助兩人面對面玩題卡
- 題目由產品方維護
- 玩家只負責選題、口頭互動、留下評語
- 紀錄保存在本機，可匯出或匯入

後續可擴充的商業方向包括：

- 專業題包
- 活動題包
- 情侶題包
- 主持人提示題包
- 遊戲結果摘要
- 認知偏差分析
- Google Drive 匯出

## 16. L1 / L2 選題互動修正

使用者在驗收 L1 / L2 介面時指出，切換 L1 關係模式時不希望 L2 題目類型跟著改變。這個情境很合理，因為玩家可能正在同一個題目類型下比較不同熟悉度的題目。

因此調整切換模式邏輯：切換 L1 時會優先保留目前選中的 L2 題目類型；只有新模式沒有該類型時，才退回可用的第一個類型。
